MySQL note-Four

Constraint

not null 非空约束

unique 唯一约束

primary key PK主键约束（既不能为null 也不能重复）

foreign key FK外键约束 。。。

check 检查约束（Oracle中支持，MySQL目前不支持）

not null

unique

字段具有唯一性，但可以为null

Primary Key

主键的作用：主键值是这行记录在这张表中的唯一标识。

一张表的主键约束只有一个，按主键字段数量，有单一主键、复合主键（多个字段联合添加一个主键约束）之分；按主键性质，有自然主键（和业务无关的自然数，推荐）、业务主键（银行卡的卡号、身份证的号码，不推荐）之分

Foreign Key

外键字段可以为null

外键引用的字段不一定是主键约束的字段，但是必须是具有唯一性的字段，因此一般常用主键约束的字段

Engine

mysql采用的默认存储引擎是InnoDB

采用的默认字符集是UTF8

MyISAM

格式文件 - 存储表结构的定义（mytable.frm）

数据文件 - 存储表行的内容（mytable.MYD）

索引文件 - 存储表上的索引（mytable.MYI）

优点：1.灵活的AUTO_INCREMENT字段处理 2.可被压缩，转换成只读表节约空间

缺点：1.不支持事务

InnoDB

表的结构存储在mytable.frm文件中

数据存储在tablespace中（一种逻辑概念），不能被压缩，或者转换为只读

优点：1.支持事务、行级锁、外键等，数据安全性最高 2.在MySQL数据库崩溃之后提供自动恢复机制

InnoDB支持级联删除和级联更新（较危险，不常用）

MEMORY

曾用名：HEPA

优点：查询速度最快

缺点：不支持事务。所有数据和索引都存储在内存中，数据容易丢失。

Transaction

一个事务是一个完整的业务逻辑单元，不可再分（多条DML语句捆绑在一起）

eg：银行账户转账，必须保证一增一减均执行成功，或者一增一减均执行失败

和事务相关的语句只有：DML语句（insert delete update）

事务特性：

A 原子性（事务是最小的工作单位，不可再分）

B 一致性（事务必须保证多条DML语句同时成功或者同时失败）

C 隔离性（事务A与事务B之间具有隔离）

D 持久性（最终数据必须持久化到硬盘文件中，事务才算成功结束）

隔离性：

1.read uncommitted 读未提交

​ 对方还未提交，我们当前事务可以读取到对方数据，存在Dirty Read（脏读现象）

2.read committed 读已提交

​ 这种隔离级别解决了脏读问题，但不可重复读

3.repeatable read 可重复读

​ 这种隔离级别解决了不可重复读的问题，但读取到的数据是幻想，是备份数据，不是最新的数据

4.serializable read 序列化读/串行化读

​ 解决了所有的问题，效率低，需要排队

Oracle数据库默认隔离级别是：read committed 读已提交

MySQL数据库默认隔离级别是：repeatable read 可重复读

并且MySQL中的事务是支持自动提交的

Index

index即索引，相当于一本书的目录

两种检索方式：

1.全表扫描 2.根据索引检索（高效率）

主键和具有unique的字段会自动添加索引，所以根据主键查询效率高。

使用模糊查询时索引失效

索引的实现原理

自动排序（可能是硬盘文件也可能是内存）

自动分区，在检索式首先定位到对应字段数据所在的分区

先定位S区，再定位M区，缩短了扫描的数量，进而快速定位到SIMTH

并且这些对应的分区都有对应的物理地址，通过物理地址定位表中的数据效率最高

索引分类

单一索引 ：给单个字段添加索引

复合索引 ：给多个字段添加索引

主键索引 ：主键上会自动添加索引

唯一索引：有unique约束的字段上会自动添加索引

视图

只有DQL语句才能以视图对象创建

对视图的增删改查会影响原表数据，但不是通过直接操作原表的方法

视图作用

视图可以隐藏表的实现细节，保密级别较高的系统，数据库只对外提供相关的视图

DBA命令

导出数据库的数据

直接在Windows的dos命令窗口执行 mysqldump nameofdatabase>D:\XXXX password

导入数据库的数据

create database nameofdatabase;

use nameofdatabase;

source 拖入.sql文件

数据库设计三范式

按照这三个范式，会尽可能地避免数据冗余

第一范式

任何一张表都应该有主键，并且每一个字段原子性不可再分。

第二范式

建立在第一范式的基础上，所有非主键字段，完全依赖主键，不能产生部份依赖。

第三范式

建立在第二范式的基础上，所有非主键字段直接依赖主键字段，不能产生传递依赖。

一对一设计

1.主键共享

A表的主键PK对应字段，是B表中主键的外键PK+FK

2.外键唯一

FK+unique
